# Signal Flow 阅读位置设计

## 目标

Signal Flow 在 TG 或 X 新消息插入顶部时，保持用户当前阅读位置稳定。用户还可以在同一设备上返回上次阅读的位置。

## 范围

- 只改 Signal Flow 前端阅读体验。
- 不改 TG/X 数据源、快照结构、消息排序、筛选规则或轮询频率。
- 桌面端和手机端都支持。

## 阅读锚点

- 以当前视口内最靠上的可见信号卡片作为阅读锚点。
- 记录卡片 ID、卡片相对视口顶部的位置和保存时间。
- 用户滚动或点击卡片后，将锚点写入当前设备的 `localStorage`。
- 持久化仅限当前设备，不同步到 VPS 或其他设备。

## 刷新补偿

- 收到实时快照或轮询快照后，列表仍按现有规则合并。
- 更新前记录当前阅读锚点相对视口顶部的位置。
- 更新后如果该卡片仍存在，计算位置偏移并补偿滚动距离。
- 结果是新消息可以继续插入顶部，但正在阅读的卡片在屏幕上的位置保持不变。
- 用户主动切换标签、时间范围、作者筛选或搜索条件时，不执行旧列表的滚动补偿。

## 返回上次阅读

- 在 Signal Flow 工具栏增加“返回上次阅读”按钮。
- 点击后定位到 `localStorage` 保存的卡片。
- 如果卡片不在当前筛选结果中，提示“上次阅读内容不在当前列表”，不改变页面位置。
- 页面打开时不自动跳转，避免妨碍用户先查看最新消息。

## 实现边界

- 抽出纯函数处理锚点选择、存储解析和滚动补偿计算，便于测试。
- 信号卡片增加稳定的 `data-signal-feed-item-id` 属性。
- 列表容器增加 `ref`，同时兼容桌面端内部滚动和手机端页面滚动。
- 使用 `requestAnimationFrame` 在 React 提交更新后执行滚动补偿。

## 测试

- 锚点存储解析能拒绝无效数据。
- 新消息插入顶部后，补偿值保持原卡片位置稳定。
- 卡片不再存在时不补偿滚动。
- Signal Flow 组件包含稳定卡片 ID、返回按钮和锚点恢复逻辑。
- 构建通过，并在本地浏览器验证桌面和手机宽度。
